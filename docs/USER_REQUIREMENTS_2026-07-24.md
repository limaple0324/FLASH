# 使用者新增需求基準（2026-07-24）

本文件整理使用者提供的兩張手寫需求，作為「輔」後續持續開發的正式輸入。
內容不得因目前尚未實作而刪除，也不得把辨識不完整的文字自行改成產品決策。

## 固定協作與驗證規則

1. 說明專案結構與模組責任；除非功能很小且高度相關，否則應拆開處理。
2. 盤點實際倉庫現況，整理各層目錄與檔案是否仍有作用。
3. SP1 視窗註冊資料應有玩家可查看的頁面或區域。
4. Windows 11 問題與相容性必須在對應里程碑做實機驗證。
5. 要跨越目前範圍、改變既定架構或開啟大型整合前，必須先討論。
6. 保留並持續驗證目前已有的組別、角色同步與進度提示能力。
7. 每次只完成一個小型、明確、可獨立驗證的區塊，測試通過後立即提交。

## 新增功能需求

1. 驗證斷線與重新連線是否可行且穩定；穩定後再加入自動辨識。
2. 提供可查看玩家習慣的頁面或區域。
3. 定義提醒區的編號與綁定規則。原稿的精確名詞仍待確認，實作前不可自行猜測。
4. 能依玩家電腦本身的規格調整或提升能力，同時保留安全上限。
5. 提醒卡的顯示／停留時間可由玩家自行設定。
6. 每個介面應有設定入口，可自訂文字與背景圖片。
7. 提供可查看目前組別全部角色資料的介面。
8. 可分角色記錄寵物或每日養成相關資訊；原稿後半句因照片裁切而不完整，
   先保留可擴充資料需求，精確欄位與觸發時機待確認。
9. 可分角色記錄「靈魂石」資訊。
10. 可分角色記錄「命魂」資訊。
11. 可分角色記錄「魂器」資訊。
12. SP4、SP5，以及目前的 SP1、SP2、SP3，均需有清楚且可分辨的程式碼邊界
    與交付紀錄；是否代表獨立套件或獨立版本，實作前需先確認，不複製出互相
    漂移的程式碼。
13. 評估並提供背包內資訊的記錄能力。
14. 驗證自動戰鬥在中斷或一個回合結束後是否可以安全接續；在安全邊界與實機
    驗證完成前，不啟用遊戲輸入。
15. 明確定義同一角色加入不同組別時的同步判定。例如 A 同時是 B 與 C 的同步
    窗口，操作 B 時 A 應依 B 的規則同步，操作 C 時 A 也應依 C 的規則同步，
    且不得造成重複操作、循環同步或互相衝突。

## 執行與停止規則

- 上述需求納入既有 SP1／SP2／SP3／SP3.5 決策與整合分支，不重新設計或推翻
  已確認的產品規則。
- 優先實作不需等待文字釐清、風險低且可單輪完成的區塊；辨識不完整的項目仍
  保留在需求中，但在確認前不寫死行為。
- 持續開發不得因完成單一區塊、一次提交或中間里程碑而停止。
- 只有使用者明確要求停止，或整個專案完成、合併、正式發布並完成回報後，才
  可以停止持續開發。
- 若單輪額度不足，只停止該輪擴張並保存進度；不得因此停用後續持續開發。

## 目前對應狀態

- 已有基礎：組別與角色資料、提醒卡資料與 30 秒生命週期、提醒卡浮層骨架、
  方案選擇與關閉流程、顯示時間保存與首頁調整入口、首頁組別角色只讀清單與
  等級／分類／定位／備註、角色資料的唯讀轉接、安全保存及啟動載入基礎、
  角色詳細資料的獨立唯讀邊界、中文呈現格式與視窗外殼、斷線／恢復邊界。
- 尚需驗證：Windows 11 實機、真正斷線重連、跨組同步衝突、安全接續。
- 尚待開發：習慣頁面、角色詳細資料頁、玩家自訂介面、角色資源與背包紀錄，
  以及 SP4／SP5 的明確邊界。
- 尚待文字確認：第 3 項的精確綁定名詞，以及第 8 項被照片裁切的欄位與觸發
  時機。這兩項不阻塞其他已清楚需求的持續開發。
