# 一、專案真正名稱與定位

## 1. 產品名稱

產品正式名稱是：

**「輔」**

FLASH 不是產品人格，也不是玩家面對的主要名稱。

FLASH 只負責：

- 啟動程式
- 承載工程架構
- 建置與更新
- 作為 GitHub（程式倉庫）名稱
- 連接遊戲與「輔」

玩家真正使用、信任、互動的對象是：

**「輔」**

因此，玩家畫面中不應一直出現：

- FLASH SP1
- 工程驗證
- 除錯資訊
- 技術模組名稱
- 開發人員才能理解的內容

## 2. 最核心定位

玩家永遠是主角。

「輔」不是主角，也不是要求玩家另外管理的一套系統。

「輔」存在的目的，是：

- 降低重複操作
- 協助處理繁瑣流程
- 記住玩家容易遺漏的事情
- 在適當時間提醒
- 在斷線或中斷時協助恢復
- 讓玩家能把注意力放回遊戲

「輔」不是單純工具。

它是一個可以長期陪伴玩家、逐漸形成默契的存在。

## 3. 最終陪伴感受

「輔」應讓玩家感受到五件事：

1. **可靠**
2. **守護**
3. **安靜**
4. **默契**
5. **信任**

這五件事最後形成的結果是：

**安心**

專案最核心的一句話：

> 讓玩家安心向前，把背後交給「輔」。

這裡的「把背後交給輔」，不是讓「輔」奪走控制權。

而是玩家知道：

- 容易遺漏的事情有人記得
- 斷線時有人注意
- 時間到了會被提醒
- 流程中斷後能繼續
- 「輔」不會擅自做玩家沒允許的事

# 二、不可推翻的產品原則

以下內容已確定，後續不能因為介面或技術方便而推翻。

## 1. 玩家擁有最後決定權

「輔」可以：

- 提醒
- 建議
- 整理
- 協助
- 自動執行已授權的重複流程

但不能：

- 擅自改變流程
- 擅自替玩家做決定
- 未經允許學習危險行為
- 因為一次特殊情況就改掉長期習慣
- 把建議變成強迫

## 2. 安心優先於效率

不是最快就一定最好。

例如：

- 不確定角色狀態時，不應亂操作
- 不確定玩家是否離開活動時，先詢問
- 無法確認視窗身分時，保持停用
- 恢復流程應先確定安全，再繼續

## 3. 不知道就不要假設

「輔」只能根據真正偵測到的資訊判斷。

不可自行捏造：

- 玩家已完成
- 玩家想做某活動
- 某角色目前在哪裡
- 某次操作一定成功
- 玩家習慣已經改變

## 4. 一致性大於聰明

玩家應該能逐漸理解「輔」的行為。

相同情況應有相同邏輯。

不能今天用一套判斷，明天又突然換另一套。

## 5. 只在有價值時出現

「輔」不需要一直說話。

應保持安靜的情況：

- 沒有新資訊
- 沒有時間壓力
- 玩家正在順利進行
- 提醒過一次且狀況沒有改變
- 顯示資訊只會增加干擾

# 三、專案與協作規則

## 1. 討論順序

所有設計應依序進行：

1. 核心理念
2. 行為
3. 資訊
4. 規格
5. 介面
6. 技術實作

不能還沒確認產品邏輯，就直接開始畫介面或寫功能。

## 2. 聊天室規則

- 一個聊天室只討論一個章節或主題
- 完成一章後，應主動指出下一章
- 視窗過長或延遲時，應主動建議更換
- 更換時必須提供一個可直接複製的完整銜接區塊
- 新聊天室不得要求使用者重新解釋已確定內容
- 不得重新設計或推翻已確認的藍圖

## 3. 分工

使用者負責：

- 提供真正遊戲流程
- 說明玩家習慣
- 判斷體驗是否合理
- 實機驗證
- 最終產品決策

系統負責：

- 技術實作
- 架構整理
- 資訊流程
- 測試
- 倉庫管理
- 建置
- 更新流程
- 問題修正

不應反覆要求使用者執行複雜終端機操作。

更新輔

# 八、SP2｜智慧與產品邏輯

SP2 回答的問題是：

> 「輔」應該如何理解玩家？

SP2 主要設計已完成。

## SP2-01｜產品哲學

核心：

- 玩家才是主角
- 「輔」是陪伴者
- 不增加玩家管理負擔
- 所有資訊都要回答玩家當下最需要知道的問題
- 自動化只是方法，不是產品目的
- 不能把玩家變成軟體管理員

## SP2-02｜工作區系統

工作區不是一般程式首頁。

它是「輔」理解玩家當下狀態後，組合出來的陪伴區域。

包含：

- 工作區概念
- 卡片式工作區
- 共用卡片
- 工作區生命週期
- 工作區邊界
- 工作區穩定性
- 工作區擴充
- 組別工作區
- 情境切換

核心原則：

> 工作區不負責管理遊戲，工作區負責理解玩家。

## SP2-03｜玩家旅程

完整旅程：

1. 開啟電腦
2. 第一時間開啟「輔」
3. 選擇組別
4. 顯示今日狀況
5. 啟動遊戲流程
6. 登入
7. 選擇角色
8. 進入日常流程
9. 遊戲中陪伴
10. 偵測活動、時間與進度
11. 中斷時恢復
12. 斷線時智慧重連
13. 完成後安靜退回待命

## SP2-04｜介面設計哲學

固定原則：

- 少文字
- 少操作
- 不打擾
- 一眼看懂
- 先看當下情境
- 玩家不需理解技術詞
- 不呈現除錯資訊
- 不讓玩家自己判讀大量數據
- 重要狀況要立即看見
- 不重要內容應收起來

## SP2-05｜提醒卡系統

提醒卡不是裝飾，也不是一般通知。

它是：

- 狀態紀錄
- 當下提醒
- 下一步提示
- 中斷恢復入口
- 快速複製操作代碼的入口

提醒卡應以最小空間，讓玩家看懂：

- 發生什麼事
- 哪一組或哪些角色
- 現在進行到哪裡
- 是否需要玩家處理
- 下一步是什麼

## SP2-06｜工作區行為

工作區的決策訊號包含：

- 玩家目前焦點
- 當前情境
- 時間機會
- 活動優先度
- 過去記憶
- 使用習慣
- 角色組別
- 中斷狀態
- 是否值得提出建議

基本決策順序：

1. 安全與斷線
2. 有時限的活動
3. 即將造成損失的事情
4. 玩家目前正在做的事情
5. 已中斷但可恢復的事情
6. 當前組別進度
7. 今日重要活動
8. 可延後活動
9. 建議
10. 保持安靜

## SP2-07｜實作規格

已定義的主要產品模組：

- 工作區
- 提醒卡
- 儀表板
- 記憶

# 九、角色與組別模型

角色不是單純名稱。

每個角色應包含：

- 角色身分
- 顯示名稱
- 等級模板
- 所屬組別
- 主號／次要／備用
- 當前視窗
- 目前活動
- 進度
- 最近狀態
- 是否中斷
- 可否恢復
- 今日已完成內容

## 組別原則

提醒與進度應優先以「組別」呈現。

原因：

如果同時開啟十四個視窗，不應跳出十四張相同提醒卡。

正確呈現例如：

```text
120 組｜守紀
5 個角色中，3 個已完成
目前第 2 次所羅門
```

# 十、狀態模型

主要狀態維持簡單：

1. 待命中
2. 執行中
3. 已完成

另有一個暫時性狀態：

```text
中斷中
```

中斷中不是最終狀態。

它只表示：

- 斷線
- 程式關閉
- 流程被打斷
- 等待恢復

恢復完成後應回到：

- 執行中
- 待命中
- 已完成

# 十一、提醒卡完整規格

螢幕右下角，工作列時間上方。

## 存續時間

一般預設三十秒。

完成後可立即消失。

## 歷史紀錄

一般提醒不需永久保存。

目前只保留：

- 斷線紀錄
- 必要恢復紀錄

## 點擊行為

可用於：

- 複製遊戲內代碼
- 開啟恢復流程
- 查看受影響角色
- 執行明確下一步

## 資訊層級

第一層：活動

```text
正在挑戰守紀
```

第二層：當前進度

```text
120 古｜第 2 次所羅門
```

第三層：日常摘要

```text
已完成 1 次
```

## 提醒卡優先度

最高：

- 斷線
- 恢復
- 時間限制
- 即將造成損失

目前已建立的活動排序基準：

1. 次元

3. 迴廊
4. 副本
5. 莊園
6. 種菜
7. 魂器
8. 守紀

此排序不是絕對寫死。

仍應受以下因素影響：

- 剩餘時間

- 玩家當前活動
- 角色重要程度
- 是否正在中斷恢復

# 十二、魔心次元組流程

組別名稱：

```text
魔心次元組
```

起點：

```text
眷顧之城
```

流程：

1. 開啟「輔」
2. 選擇魔心次元組
3. 幻魔修練第一次
4. 莊園種材料
5. 領取迴廊
6. 到凋零孤村接次元空間
7. 次元空間掛機約一小時
8. 幻魔修練第二次
9. 農場種材料
10. 等待作物成熟
11. 到競技場執行魔物守紀

## 幻魔修練

- 每十分鐘可領取
- 一般一天三次
- 前段一次
- 次元後一次
- 晚間一次
- 最後到龐貝城換金票

## 次元

- 主要任務是偵測斷線
- 玩家開機後通常會直接銜接次元
- 不需過度提醒
- 離開次元空間可判定活動結束

## 農場

- 成熟時間約四十分鐘
- 四十五分鐘為容錯
- 每個視窗獨立計時
- 延誤十分鐘後提高優先度
- 延誤十五分鐘可能已造成損失
- 即使已延誤，仍應提醒玩家繼續處理

## 守紀

- 每個角色每天十六次
- 進入競技場後視為執行中
- 仍在競技場就不應自行判斷取消
- 離開競技場時，前幾次先詢問是否取消
- 後續可依穩定習慣調整
- 十四個視窗全離開競技場後，可判定組別守紀結束

## SP3-03｜情感回饋

情感回饋不是稱讚玩家。

**理解玩家。**

回饋原則：

- 時間準確

- 不重複轟炸
- 不使用冰冷系統通知語氣
- 不過度熱情
- 不搶走玩家注意力

信任來自：

- 準確
- 穩定
- 尊重
- 可靠

## SP3-04｜陪伴個性

「輔」的個性已確定為：

- 可靠
- 守護
- 安靜
- 默契
- 信任

不是討好型。

不是一直聊天型。

不是命令型。

不是誇張情緒型。

「輔」應像一個站在玩家背後、知道何時出手，也知道何時保持安靜的夥伴。

# 二十、SP3.5｜決策哲學

SP3.5 是 SP3 與 SP4 之間的橋梁。

目的：

> 在正式進入更深的整合前，先確定「輔」遇到未知狀況時應如何思考。

## SP3.5-01｜決策原則

此章已完成。

五大原則：

1. 永遠站在玩家這一邊
2. 安心優先於效率
3. 玩家永遠擁有最後決定權
4. 不知道就不要假設
5. 一致性大於聰明

總原則：

> 每一個決策都應讓玩家更安心，而不是更焦慮。

## SP3.5-02｜決策驗證

此章尚未完整討論，是下一個正式設計章節。

需要回答：

### 1. 如何判斷一次決策成功

例如：

- 玩家接受建議
- 玩家完成下一步
- 玩家沒有被打擾
- 沒有造成誤操作
- 流程成功恢復
- 提醒時機正確

### 2. 什麼情況表示需要修正

例如：

- 玩家經常關閉同類提醒
- 玩家每次都選擇不同處理
- 提醒太早或太晚
- 建議增加操作負擔
- 習慣判斷與真實行為不符

### 3. 如何避免一次特殊事件改變長期習慣

需要區分：

- 單次例外
- 短期改變
- 穩定新習慣

### 4. 如何在一致性下持續改善

改善不能讓玩家覺得「輔」突然變成另一套系統。

### 5. 新增判定因素的原則

任何新判定因素都必須：

- 有真實資料來源
- 能解釋原因
- 不越過玩家權限
- 可被取消
- 不破壞既有核心
