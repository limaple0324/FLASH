# Integration First Batch

第一批只整合低風險、可測試、無遊戲操作副作用的項目。

## 執行順序

1. 產品名稱與版本語意
   - 主名稱固定為「輔」。
   - FLASH 保留為啟動媒介與技術代號。
   - 不改動既有啟動流程與自我檢查。

2. 狀態資料模型
   - 建立待命中、執行中、已完成三態。
   - 使用 Enum，避免任意字串造成狀態分裂。
   - 先建立單元測試，不接 UI。

3. 角色資料模型
   - 角色唯一身分。
   - 角色名稱、等級、角色重要度。
   - 暫不改動現有 Window Registry 格式。

4. 組別資料模型
   - 一個組別包含多個角色。
   - 卡片與工作區以組別為主要呈現單位。
   - 不在第一批加入跨組自動切換。

5. 活動資料模型
   - 活動名稱、類型、重置規則、最大次數、適用角色。
   - 不把遊戲內活動細節寫死在 UI。

6. 活動進度接口
   - 支援目前次數、狀態、開始時間、完成時間。
   - 預留每日 00:00 重置。
   - 第一批只處理資料，不執行遊戲操作。

7. 服務註冊
   - 新模型與 Store 穩定後，才接入 AppContext。
   - `main.py` 只負責組裝，不承擔決策。

## 第一批完成標準

- 每個模型皆有型別與驗證規則。
- 每個模型皆有單元測試。
- 不破壞現有 SP1 測試。
- 不送出任何遊戲輸入。
- 不直接改造現有主畫面。
- 每一項獨立提交並在 Integration Board 標記 ✅。

## 延後項目

- 工作區畫面。
- 卡片右下角浮層。
- 活動優先度。
- 習慣學習。
- 陪伴文案。
- 斷線後自動恢復操作。

這些項目在第一批資料骨架與測試完成後依 Sprint 順序接入。
