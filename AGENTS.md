# 「輔」專案規則

本檔適用於整個倉庫。開始工作前，先讀取：

1. `PROJECT_CONTEXT.md`
2. `docs/USER_REQUIREMENTS_2026-07-24.md`
3. `docs/00_輔_專案完整總整理與未來路線圖.md`
4. `docs/INTEGRATION_BOARD.md`
5. `docs/INTEGRATION_ARCHITECTURE_BASELINE.md`
6. `docs/INTEGRATION_BLUEPRINT_MAP.md`
7. `SP1_VERIFICATION.md`

產品決策以正式文件為準；程式是否完成，必須共同核對目前分支的原始碼、測試、
提交紀錄與自我檢查，不可只看文件勾選框。

## 固定產品規則

- 正式產品名稱是「輔」；FLASH 只作為倉庫與工程承載名稱。
- 玩家是主角；「輔」應可靠、守護、安靜、有默契且值得信任。
- 不重新設計或推翻已確認的 SP1、SP2、SP3、SP3.5。
- 不擅自改產品名稱、固定文案、捷徑或圖示。
- 固定圖示是藍底白色加號，不可有黑框、黑底、白色虛線或多餘線條。
- 玩家擁有最後決定權；不知道的資料不可猜測。
- 使用者需求第 3、8 項文字不完整，確認前不得補造固定行為。
- 所有玩家操作失敗都要有中文畫面提示或可查看的中文紀錄。
- 不向玩家顯示角色識別碼、視窗控制資料或其他工程內部資訊。

## 架構與安全邊界

- 維持 SP1 Engine、SP2 Brain、SP3 Presentation、SP3.5 Companion 的責任邊界。
- 沿用 `AppContext`、`EventBus`、`PathManager` 與統一 Logger，不另建平行系統。
- 不更改 SP1 已固定的 Recovery、Smart Reconnect、External Adapter 契約。
- 不把 SP2 決策放進 Adapter，不把 SP3 文案放進 Recovery／Reconnect 核心，
  不讓 UI 自行修改領域狀態。
- 在安全邊界與 Windows 11 實機驗證完成前，不啟用真正遊戲輸入。
- 不把真正斷線重連、背景輸入或 Windows 實機能力標示為已完成，除非已有實機證據。

## 工作與 Git 規則

- 最新功能開發以 `integration/sp2-sp3-sp35` 為基準；`main` 不是目前最新功能成果。
- 取得遠端最新狀態後才判斷分支與提交；不得覆蓋既有未提交修改。
- 每次只完成一個小型、清楚、可獨立驗證的區塊。
- 每個區塊完成後執行相關測試；可行時也執行全測試、原始碼編譯與完整自我檢查。
- 驗證通過後，明確提交該區塊並推送；不得混入無關修改。
- 完成一個區塊或一次執行，不代表整個專案完成或停止。
- 大型分支整合、架構改變或跨越目前範圍前，先向使用者說明。
- `main` 與整合分支有歷史分岔；不得強推、硬合併或以任一方整檔覆蓋衝突。
- 玩家更新入口只保留單一「更新輔」，不得恢復每十五分鐘 Git 同步排程。

## 版本與回報規則

- 回報必須分開標示：設計已定案、程式已實作、自動測試已通過、
  Windows 成品已建置、Windows 11 實機已驗收、正式發布。
- SP1、SP2、SP3 各自要有可辨識的獨立交付檔案，另有一份整體彙整檔。
- SP2、SP3 是建立在前一層之上的累積成果；不得複製成三套或四套互相漂移的原始碼。
- 獨立交付檔案的類型、版本與提交來源必須可追溯，且不得把整合版誤標成 SP1 成品。
- 未完成 Windows 11 實機驗證前，只能標示為程式／自動測試或工程建置狀態。
